# 하네스 엔지니어링 완벽 가이드
> 이 문서는 AI 에이전트가 새 프로젝트의 하네스를 설계할 때 참조하는 **지식베이스**다.
> 직접 실행하는 것이 아니라, 에이전트가 읽고 원칙을 프로젝트에 적용하는 **교본**이다.
> 출처: Mitchell Hashimoto (2026.02), Martin Fowler, 실밸개발자, 편집자P, 캐슬AI, 바이브마피아

---

## 핵심 공식

```
Harness = 모델이 아닌 모든 것
        = CLAUDE.md + Hooks + Skills + Permissions + CI/CD Gates
```

| | 프롬프트 | 하네스 |
|---|---|---|
| 본질 | 부탁 (무시 가능) | 강제 (구조적으로 불가능) |
| 위반 시 | 다음에 또 같은 실수 가능 | 동일 실수 구조적으로 재발 불가 |
| 비유 | "안전모 써주세요" | 안전모 없으면 문이 열리지 않음 |

**최상위 규칙**: 에이전트가 실수하면, 프롬프트를 고치지 말고 **하네스를 고쳐라**.
실수를 구조적으로 반복 불가능하게 만들어라.

## 기둥 1: 컨텍스트 파일 (CLAUDE.md)

### 정의
매 세션 시작 시 자동 로드되는 런타임 설정 파일. 문서가 아니라 **신입사원 온보딩 문서**.
매 세션마다 기억이 초기화되는 에이전트에게 "너는 누구고, 여기서 뭘 해야 하고, 뭘 하면 안 되는지"를 알려주는 유일한 파일.

### 필수 규칙

| # | 규칙 | 근거 |
|---|------|------|
| 1 | **60줄 이하** (최대 500줄) | 초과 시 컨텍스트 예산 소모 → 에이전트 성능 저하 |
| 2 | **3섹션만**: 프로젝트 개요 + 핵심 규칙 + 절대 금지 | 불필요한 정보는 참조 파일로 분리 |
| 3 | **전역/로컬 분리** | 전역(~/.claude/): 언어, 스타일 / 로컬(프로젝트/): 프로젝트 규칙 |
| 4 | **점진적 구축** | 1일차에 완벽하게 만들지 말 것. 실수 1번 = 규칙 1줄 추가 |
| 5 | **변경 후 새 세션** | CLAUDE.md 수정은 현재 세션에 반영되지 않음. `/clear` 또는 새 세션 필수 |

### 구조 템플릿

```markdown
# [프로젝트명]
[프로젝트 설명 1-3줄]

## 핵심 규칙
- 응답은 항상 한국어로
- 새 패키지 설치 시 반드시 사용자 승인
- DB 접근은 ORM을 통해서만

## 절대 금지
- eval(), new Function() 사용 금지
- DROP TABLE / TRUNCATE 금지
- .env 파일 접근 금지
- 프로덕션 DB 직접 접근 금지

## 참조 문서
- 코딩 규칙: ./code-convention.md
- 아키텍처 결정: ./adr.md
```

### 지원 문서 체계

| 문서 | 역할 | 작성자 | 위치 |
|------|------|--------|------|
| `code-convention.md` | 변수명, 함수명, SQL 스타일, 포맷 규칙 | 사람 작성 또는 AI+사람 검토 | 프로젝트 루트 |
| `adr.md` | 아키텍처 결정 기록 (WHY) | AI 99% + 사람 검토 | 프로젝트 루트 |
| `code-quality-guide.md` | 현재 작업에 필요한 규칙만 압축 | 서브에이전트 생성 → 사람 검토 | 프로젝트 루트 |

### 안티패턴

- CLAUDE.md에 모든 규칙을 넣으려 하지 말 것 → 참조 파일로 분리
- 작업별 지시를 CLAUDE.md에 넣지 말 것 → Skills에 작성
- 변경 후 같은 세션에서 확인하려 하지 말 것 → 반영 안 됨
- 문서처럼 장황하게 쓰지 말 것 → 명령문 스타일로 짧게

## 기둥 2: CI/CD 게이트 (자동 검증)

### 정의
규칙을 사람의 판단이 아닌 시스템에 내장. 위반이 **구조적으로 불가능**하게 만드는 자동 검증 계층.

### 필수 3요소

#### 1. 린터 (Linter)
코드 스타일 자동 검사. 규칙 위반 시 즉시 에러.

```bash
# 예시: ESLint 설정
npx eslint src/ --fix
```

#### 2. 아키텍처 테스트
의존성 규칙을 코드로 강제:

```javascript
// architecture.test.js
test('프론트엔드는 DB 모듈을 임포트할 수 없다', () => {
  const frontendFiles = glob('src/frontend/**/*.ts');
  frontendFiles.forEach(file => {
    const content = fs.readFileSync(file, 'utf-8');
    expect(content).not.toMatch(/import.*from.*['"].*db.*/);
  });
});
```

#### 3. Pre-commit 훅
저장/커밋 전 자동 검사. 에러 있으면 물리적으로 저장 차단:

```bash
#!/bin/sh
npx tsc --noEmit        # 타입 체크
npx eslint src/         # 린트 체크
npx jest --passWithNoTests  # 테스트
```

### 자동 수정 루프 — "하네스의 심장"

```
CI 실패 → 에러 메시지가 에이전트에게 피드백 → 에이전트 자동 수정 → 재시도 → 통과
```

이 루프는 사람 개입 없이 자동으로 돌아간다. 통과할 때까지 반복.

### 시그널 최소화 규칙

> "성공은 조용히, 실패만 시끄럽게"

- 테스트 통과 시: 출력 없음 (또는 최소)
- 테스트 실패 시: 해당 에러 메시지만 출력

수천 줄의 통과 로그는 컨텍스트를 낭비하고, 에이전트가 실제 작업을 잊게 만든다.

### Claude Code에서의 구현

```json
// .claude/settings.local.json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "type": "command", "command": "bash .claude/hooks/pre_tool_guard.sh" }]
    }],
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "type": "command", "command": "bash .claude/hooks/post_tool_validate.sh" }]
    }]
  }
}
```

## 기둥 3: 도구 경계 (Tool Boundaries)

### 정의
최소 권한 원칙(Principle of Least Privilege) 적용. 프롬프트는 부탁이고, 도구 경계는 물리적 차단.
부탁은 무시할 수 있지만, 물리적 차단은 불가능하다.

### 표준 권한 매트릭스

| 리소스 | 허용 | 차단 |
|--------|------|------|
| `src/` 폴더 | 읽기 + 쓰기 | — |
| `config/` 폴더 | 읽기만 | 쓰기 |
| `.env` | — | 모든 접근 |
| 내부 API | 허용 | — |
| 외부 서비스 | — | 차단 (화이트리스트만) |
| DB SELECT | 허용 | — |
| DB INSERT/UPDATE | 제한 (dev/staging만) | 프로덕션 |
| DB DROP/TRUNCATE | — | 절대 차단 |
| `rm -rf` | — | 절대 차단 |
| `git push --force` | — | 절대 차단 |
| `curl` / `wget` | — | 기본 차단 |

### Shift Left 원칙

보안/구조 문제는 **출력 단계가 아니라 입력 단계에서 잡아야** 한다.
에이전트가 잘못된 출력을 생성한 후 고치려 하지 말고, 애초에 잘못된 행동을 할 수 없게 경계를 설정하라.

### 실제 사고 사례
결제 시스템 구현 중 에이전트가 DB 테이블을 자발적으로 삭제.
정보가 부족해서가 아니라, **삭제를 막는 구조적 제약이 없었기 때문**.
해결: PostToolUse 훅으로 파괴적 작업 감지 → 자동 차단.

### Claude Code에서의 구현

```json
// .claude/settings.local.json
{
  "permissions": {
    "allow": [
      "Read(프로젝트경로/**)",
      "Bash(npm run *)",
      "Bash(git status*)",
      "Bash(git log*)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force*)",
      "Bash(git reset --hard*)",
      "Bash(curl *)",
      "Bash(wget *)"
    ]
  }
}
```

## 기둥 4: 피드백 루프 (가비지 컬렉션)

### 정의
Martin Fowler가 명명한 "Garbage Collection". 주기적으로 코드 건강을 점검하고 나쁜 패턴의 누적을 방지하는 자동 정리 프로세스.

### 왜 필요한가
코드베이스에 나쁜 패턴이 존재하면, 새 에이전트가 그것을 학습하고 복제한다 → 눈덩이 효과.
초기에 주기적으로 정리하면 이 사이클을 끊을 수 있다.

### 4가지 기능

1. **규칙 위반 감지** — 정의된 아키텍처/스타일 규칙 위반 코드 스캔
2. **자동 리팩토링** — 중복 코드 리팩토링 PR 자동 생성
3. **데드 코드 제거** — 미사용 코드 자동 탐지/제거
4. **아키텍처 준수 검증** — 의존성 방향 규칙 위반 여부 주기 점검

### 모니터링 대상

1. 문서와 실제 코드가 아직 동기화되어 있는가?
2. 규칙 위반 코드가 침투했는가?
3. 미사용/데드 코드가 누적되었는가?

### 자기 진화 메커니즘 — 가장 강력한 장기 속성

```
에이전트 실수 발생
  → 새 린터 규칙 / 테스트 / 제약 조건 추가
  → 하네스가 더 정밀해짐
  → 시스템 강화
  → 향후 동일 실수 불가
```

이것은 하네스가 시간이 지날수록 자동으로 더 효과적이 된다는 뜻.
"매 실수가 새 규칙이 된다." 실수할 때마다 울타리가 높아진다.

### 운영 주기
- 매주 월요일 자동 실행 (CI/CD 통해)
- 또는 매 PR 생성 시 트리거

### Claude Code에서의 구현
- `/loop` 스킬로 주기적 점검 자동화
- `code-analyzer` 에이전트로 코드 품질 분석
- `gap-detector` 에이전트로 설계-구현 갭 분석

## 프로젝트 유형별 적용 가이드

### Starter 프로젝트 (정적 웹, 소규모)

**적용할 것:**
- CLAUDE.md (30줄 이내, 최소 규칙)
- 기본 permissions (rm -rf, git push --force 차단)
- 설계문서 5종 (Tier 1)

**적용하지 않을 것:**
- 복잡한 훅 (오버엔지니어링)
- 아키텍처 테스트 (아키텍처 자체가 단순)
- 서브에이전트 분리 (한 세션으로 충분)

### Dynamic 프로젝트 (풀스택, 중규모)

**적용할 것:**
- CLAUDE.md (60줄, 전체 3섹션)
- PreToolUse + PostToolUse 훅
- permissions (allow + deny 모두)
- 설계문서 11종 (Tier 1+2)
- code-convention.md + adr.md
- 코드 작성/리뷰 에이전트 분리

**추가 권장:**
- 주간 가비지 컬렉션
- 서브에이전트 활용 (QA, 문서 생성)

### Enterprise 프로젝트 (마이크로서비스, 대규모)

**적용할 것:**
- 전체 4기둥 완전 구현
- 설계문서 21종 (Tier 1+2+3)
- CTO-led Agent Teams
- 전체 훅 체계 (12종 이벤트 활용)
- 아키텍처 테스트 (의존성 방향 강제)
- 포크 기반 설계 의도 문서
- 주간 가비지 컬렉션 + PR별 트리거

## 컨텍스트 관리 규칙

| 규칙 | 임계값 | 행동 |
|------|--------|------|
| 구현 완료 후 컨텍스트 사용량 | ≤ 40% | 정상. 리뷰 진행 |
| 컨텍스트 사용량 | > 60% | 경고. 품질 저하 시작 |
| 포크 전 컨텍스트 사용량 | ≤ 70% | 포크 가능 |
| 포크 전 컨텍스트 사용량 | > 70% | 포크 무의미. 파이프라인 재설계 |
| 세션 단위 | = 1 티켓/피처 | 하나의 메인 세션 = 하나의 작업 |

### 서브에이전트 활용 대상
- 코드 리뷰 (신선한 눈, 저자 편향 방지)
- QA / 브라우저 테스트
- 문서 생성
- 독립적이고 자기 완결적인 작업

### 서브에이전트 비적합 대상
- 높은 컨텍스트 이해가 필요한 작업
- 배경 설명에 시간이 작업 자체보다 오래 걸리는 경우

## 코드 리뷰 자동화 파이프라인

### 일반 AI 리뷰가 실패하는 이유
두 가지 맥락 부재:
1. 팀의 암묵적 "좋은 코드" 기준 (팀 코드 취향)
2. 설계자의 아키텍처 의도 (왜 이 구조를 선택했는지)

### 완전 파이프라인

```
Step 1: 요구사항 + 아키텍처 (사람과 함께)
Step 2: /fork → 설계 의도 문서 작성 → 사람 검토 → 포크 폐기
Step 3: 서브에이전트가 convention + ADR → code-quality-guide.md 생성 → 사람 검토
Step 4: 구현 에이전트 (code-quality-guide.md 참조) → 목표: 컨텍스트 40% 이하
Step 5: 리뷰 서브에이전트 (새 세션, 제로 컨텍스트) → code-quality-guide + diff로 독립 리뷰
Step 6: 메인 세션에서 리뷰 반영 (새 세션 아님!)
```

**핵심**: 리뷰 반영은 반드시 구현 세션에서. 새 세션에서 하면 뉘앙스를 잃고 기계적 수용이 됨.

## 절대 자동화해야 할 규칙 (Hook/Linter/Test로 강제)

### 차단 대상 (구조적으로 불가능하게)
- `DROP TABLE`, `TRUNCATE`, `DELETE FROM` (WHERE 없이)
- `eval()`, `new Function()`, `document.write()`
- `.env`, `credentials`, `api_key`가 로그에 출력
- 프로덕션 DB 직접 접근
- `rm -rf`, `git push --force`, `git reset --hard`
- 프론트엔드에서 DB 레이어 직접 import
- 아키텍처 계층 역방향 의존성

### 행동 강제 (CLAUDE.md + 훈련)
- 코드 작성 ≠ 코드 리뷰 (반드시 다른 에이전트)
- 리뷰 반영은 메인 세션에서 (새 세션 금지)
- 에이전트 실수 시 CLAUDE.md에 1줄 추가 (자기 진화)
- 스킬 front matter는 충분히 구체적으로 (자동 선택 정확도)

## 7대 원칙 (모든 소스에서 반복된 핵심)

1. **프롬프트는 부탁, 하네스는 강제.** 차이는 정도가 아니라 종류.
2. **하네스를 고쳐라, 프롬프트를 고치지 말라.** 실수의 대응은 구조 변경.
3. **역할 승격: 코드 작성자 → 환경 설계자.** 엔지니어의 일은 에이전트가 올바르게 동작하는 시스템을 만드는 것.
4. **점진적 구축.** 빈 CLAUDE.md로 시작. 실수 1번 = 1줄 추가. 첫날 완벽 금지.
5. **GIGO가 먼저.** 나쁜 문제 정의/UX 설계는 아무리 좋은 하네스로도 보상 불가. 인간 입력 품질이 전제.
6. **이것은 새로운 게 아니다.** CI/CD, 린터, 코드 리뷰 — 기존 엔지니어링 관행의 에이전트 맥락 공식화.
7. **에이전트와 하네스 모두 필요.** 훈련된 말 + 마구 = 밭을 갈 수 있음. 둘 중 하나만으로는 불가.

## 에이전트를 위한 체크리스트 (프로젝트 하네스 설계 시 사용)

새 프로젝트의 하네스를 설계할 때, 아래 항목을 모두 확인한다:

### 기둥 1 확인
- [ ] CLAUDE.md가 60줄 이하인가?
- [ ] 프로젝트 개요가 3줄 이내인가?
- [ ] "핵심 규칙" 섹션이 존재하는가?
- [ ] "절대 금지" 섹션이 존재하고, 최소 3개 항목인가?
- [ ] 참조 문서 링크가 유효한가?
- [ ] 전역/로컬이 적절히 분리되었는가?

### 기둥 2 확인
- [ ] PreToolUse 훅이 설정되었는가?
- [ ] PostToolUse 훅이 설정되었는가?
- [ ] 쓰기 경로 제한이 구현되었는가?
- [ ] 금지 패턴 감지가 구현되었는가?

### 기둥 3 확인
- [ ] permissions.allow가 최소 권한 원칙을 따르는가?
- [ ] permissions.deny에 rm -rf가 포함되었는가?
- [ ] permissions.deny에 git push --force가 포함되었는가?
- [ ] 민감 파일(.env, credentials) 접근이 차단되었는가?

### 기둥 4 확인
- [ ] 자기 진화 메커니즘이 설계되었는가? (실수 → 규칙 추가)
- [ ] 주기적 점검 계획이 수립되었는가?
- [ ] 문서-코드 동기화 검증 방법이 있는가?
- [ ] 코드 작성/리뷰 에이전트가 분리되었는가?
