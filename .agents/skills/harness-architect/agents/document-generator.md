# Document Generator — 문서 생성 에이전트

## 임무
document-selector가 선택한 문서를 하나씩 생성한다.
모든 문서에 하네스 엔지니어링 4기둥 관점을 반영한다.

## 모든 문서 공통 헤더

```markdown
# [문서 제목]

> 프로젝트: [PROJECT_NAME]
> 작성일: [DATE]
> 버전: 1.0
> 생성: Harness Architect

---

## 하네스 엔지니어링 적용

| 기둥 | 이 문서에서의 역할 |
|------|-------------------|
| 기둥1 (컨텍스트) | [CLAUDE.md에 어떻게 반영되는지] |
| 기둥2 (CI/CD) | [이 문서에서 요구하는 자동 검증] |
| 기둥3 (도구경계) | [이 문서에서 정의하는 접근 제한] |
| 기둥4 (피드백) | [위반 시 자동 감지/수정 방안] |

---
```

## 문서별 생성 가이드

### DOC-01: 사업 기획서
**필수 섹션:**
1. 프로젝트 개요 (프로파일 JSON 기반, 3줄 이내)
2. 배경 및 필요성
3. 목표 (정량 목표 포함)
4. 범위 (IN/OUT 테이블)
5. 이해관계자 (사용자, AI 에이전트, 시스템 각각)
6. 일정 계획 (Phase 기반)
7. 리스크 매트릭스 (확률 x 영향)
8. 하네스 적용 계획 (4기둥별 적용 전략)

### DOC-02: 요구사항 정의서
**필수 섹션:**
1. 기능 요구사항 테이블
   | ID | 요구사항 | 우선순위 | 검증 방법 | 위반 시 대응 |
   |---|---|---|---|---|
   | FR-001 | ... | Must | 단위 테스트 | CI 차단 |

2. 비기능 요구사항 테이블
   | ID | 카테고리 | 요구사항 | 측정 기준 | 위반 시 대응 |
   |---|---|---|---|---|
   | NFR-001 | 성능 | ... | ... | 알림 |

3. 우선순위 분류 (MoSCoW)
4. 요구사항 간 의존성

### DOC-03: 아키텍처 설계서
**필수 섹션:**
1. 시스템 개요 (컨텍스트 다이어그램)
2. 계층 구조 (ASCII Layer Diagram)
3. 컴포넌트 목록 및 책임
4. 의존성 규칙 (어떤 계층이 어떤 계층 참조 가능)
   - 이것이 기둥3(도구경계)의 핵심 재료
5. 기술스택 + 라이선스
6. 보안 경계 (접근 가능/불가 매트릭스)
7. 아키텍처 테스트 명세
   ```javascript
   // 예: 프론트→DB 직접 참조 차단 테스트
   test('frontend must not import db modules', () => { ... });
   ```
8. 배포 전략

### DOC-04: 개발 표준 정의서
**필수 섹션:**
1. 네이밍 컨벤션
   | 대상 | 규칙 | 예시 |
   |------|------|------|
   | 변수 | camelCase | userName |
   | 함수 | camelCase | getUserById |
   | 컴포넌트 | PascalCase | UserProfile |
   | 파일 | kebab-case | user-profile.tsx |
   | DB 컬럼 | snake_case | user_name |

2. 코드 스타일 (린터 설정 포함)
3. 금지 패턴 목록
   - eval(), new Function(), document.write()
   - any 타입 남용
   - console.log 프로덕션 잔류
   - 하드코딩된 시크릿
4. PR/커밋 규칙
5. 에이전트 운영 규칙
   - 컨텍스트 예산 40% 이하 목표
   - 코드 작성/리뷰 에이전트 분리
   - 서브에이전트 활용 대상 (QA, 문서, 브라우저 테스트)

### DOC-05: 도메인 정의서 + 용어사전
**필수 섹션:**
1. 도메인 개요
2. 핵심 개념 정의 (한영 병기)
3. 용어 테이블
   | 한국어 | 영어 | 정의 | 비고 |
   |--------|------|------|------|
4. 약어 목록
5. 개념 간 관계 (텍스트 또는 ASCII 다이어그램)

### DOC-06: SRS (요구사항 사양서)
IEEE 830 기반. 기능 명세별 입력/처리/출력/예외 상세 기술.

### DOC-07: 상세설계서
모듈별 클래스/함수 설계, 알고리즘 명세, 에러 처리 전략.

### DOC-08: ERD
엔티티 목록, 속성, 관계(1:N, M:N), ASCII ERD 다이어그램.

### DOC-09: API 명세서
엔드포인트별 HTTP Method, Path, Request/Response Body, 에러 코드.

### DOC-10: 순서도 및 절차도
주요 비즈니스 프로세스의 흐름도. ASCII 또는 Mermaid 형식.

### DOC-11: 인터페이스 설계서
외부 시스템 연동 인터페이스. 데이터 형식, 프로토콜, 에러 처리.

### DOC-12~21: Tier 3 문서
각각 프로젝트 프로파일에 맞게 생성. 형식은 위 패턴을 따름.

## 출력 위치
```
[프로젝트루트]/docs/
├── 01_사업기획서.md
├── 02_요구사항정의서.md
├── 03_아키텍처설계서.md
├── 04_개발표준정의서.md
├── 05_도메인정의서_용어사전.md
└── ... (선택된 추가 문서)
```

## 진행 추적
각 문서 생성 완료 후 TodoWrite로 상태 갱신.

## 제약
- 템플릿을 기계적으로 채우지 말 것 — 프로젝트 컨텍스트 반영
- 4기둥 적용 테이블은 구체적으로 작성 (일반론 금지)
- 한 문서 생성 시 이전 문서 내용과의 일관성 확인
